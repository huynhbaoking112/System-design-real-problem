Part 1

API Endpoint Design

The system will feature API endpoints designed for submitting and displaying comments, responding to comments, and utilizing moderation tools. Each of these endpoints will have a specific request and response format.

POST /comment

Request Body

{
  "content": "" // the content of comment
}

Response Body

{
  "status": "success" | "failed",
  "message": "", // string
  "comment": {
    "id": "generated_id",
    "content": "the-content-of-the-content",
    "posterId": "the-user-id-of-poster",
    "posterName": "The display name of poster",
    "postTime": "2022-01-01T12:00:00Z"
  }
}

GET /{topic_id}/comments?cursor={last_timestamp}&size={size}

This API endpoint returns data sorted in descending order, from the most recent to the oldest.

Path Variables

  • topic_id: The variable topic_id is typically associated with a specific page in a one-to-one relationship. By inputting the relevant topic_id of the page, you can retrieve the comments related to that page.

Request Parameters

  • last_timestamp: This parameter is used to query the timestamp of the last comment from the previous list. If you're retrieving data from the first page, this parameter should not be set.
  • size: This parameter determines the number of lists retrieved at once. The default value is 20, but it can be set to a maximum of 50.

Part 2

High-Level Design

The Comment Service is designed to execute essential business operations using a stateless approach. This design allows for easy cluster expansion deployment, making it an ideal solution for handling high levels of concurrency.

  • The client sends a request to the server.
  • Upon receiving a request, the server first directs it through a load balancer before it reaches the Comment Service. This ensures efficient distribution of network traffic.
  • For retrieval operations, the Comment Service initially attempts to fetch data from the Cache. If successful, the data is read directly from the Cache.
  • If the Cache does not contain the requested data, the Comment Service then queries the Comment Database. The result is subsequently returned to the client and simultaneously written back into the cache for future reference.
  • Write operations require an initial request to the User Authentication Service to validate permissions. This step ensures that only authorized users can modify the data.
  • If the user does not have the necessary permissions, the request is denied and the client is notified.
  • Once the user's permissions have been verified and approved, modifications can be made to the Comment Database.

Part 3

Detailed Design

Data Store

Database Type

We propose using either SQL or NoSQL databases for the Comment Database, with a preference for NoSQL databases due to their high performance, high availability, and easy scalability. For the purpose of this demonstration, we will use MongoDB, a popular NoSQL database.

For the User Database, we recommend using a Relational Database Management System such as PostgreSQL or MySQL. These databases are adept at handling structured data and offer strong consistency, a critical factor for user authentication.

Data Schema

The Comment Database will comprise topics, comments, upvotes and downvotes collections. The topics collection will include the following fields:

{
  "_id": ObjectId("..."),
  "topic_id": "the-topic-id",
  "total_count": 12345, // The total count of comments of the topic.
}

The comments collection as below:

{
  "_id": ObjectId("..."), // as comment_id
  "content": "The content of comment",
  "user_id": "user_id", // indexed
  "topic_id": "topic_id", // indexed
  "replied_comment_id": "",
  "upvote_count": 123,
  "downvote_count": 12,
  "timestamp": ISODate("...")
}

The upvotes collection as below:

{
  "_id": ObjectId("..."),
  "topic_id": "topic_id",
  "comment_id": "comment_id", // indexed
  "user_id": "user_id", // indexed
  "timestamp": ISODate("...")
}

The downvotes collection as follows:

  "topic_id": "topic_id",

Database Partitioning

To handle the large volume of data, we can use horizontal partitioning (sharding). We can shard the comments, upvotes and downvotes collections by topic_id. This way, data modification operations, such as upvotes and downvotes, are transactionally controlled on a single partition, with data from different topics spread across multiple nodes, reducing the load on each node and improving performance.

Database Replication

To ensure high availability and data durability, we can use database replication. Each write operation is performed on the primary database and then replicated to one or more secondary databases. If the primary database fails, one of the secondary databases can take over.

MongoDB supports replication natively through replica sets. The primary node receives all write operations, and all other nodes are secondary nodes that replicate the primary node's data set.

By default, MongoDB reads and writes are on the primary node. We can set the read preference to secondary to reduce read pressure, accepting briefly outdated data because the Comment System can rely on eventual consistency.

Data Retention and Cleanup

Based on the requirement for 10 years of data retention, we need to implement a data cleanup strategy. MongoDB supports Time-To-Live indexes, which can automatically delete expired data from the database. 

use commentDatabaseName; // Specify comments database name
db.comments.createIndex({ "timestamp": 1 }, { expireAfterSeconds: 315360000 })

Cache

To improve read performance and reduce database pressure, we can use a cache like Redis. Comments related to a specific topic typically do not change often, so after each database query, the data can be stored in cache with a short TTL.

If the client requests the same content within the TTL timeframe, the Comment Service retrieves the data from cache and avoids an extra database query. Once the TTL expires, the cache is cleared and the database is queried again.

Analytics

To track user behavior and system performance, we can use an analytics system like Google Analytics or a custom-built analytics system.

  • User Behavior Metrics: Number of comments posted, number of upvotes/downvotes, number of replies, etc.
  • System Performance Metrics: Response time, error rates, cache hit/miss ratio, etc.
  • Business Metrics: Daily active users, user growth rate, user retention rate, etc.

The analytics data can be stored in a separate analytics database, such as Redshift or BigQuery. Real-time analytics tools like Apache Kafka or Amazon Kinesis can process and analyze data in real time.

Part 4

Follow-Up Detailed Design Questions and Answers

How to Cache Pagination Data?

The interface GET /{topic_id}/comments?cursor={last_timestamp}&size={size} is utilized to paginate the comment queries for a specific topic. The results are stored in cache using a combination of topic_id, last_timestamp, and size as the key. 

{topic_id}:{last_timestamp}:{size}
{topic_id}:{0}:{size} # Set to 0 when last_timestamp is not available

example-topic1:1692278954682:20
# Here, topic_id is "example-topic1"
#     last_timestamp is "1692278954682"
#     size is "20"

If we need to clear the cache for a specific topic, we can locate and delete based on the first part of the key, which is the topic_id.

How to Avoid Duplicate Data When Switching Page?

This issue happens when new data is inserted while a client is viewing paginated data, leading to duplicate data at the intersection of two pages.

This is avoided in the design of the GET /{topic_id}/comments?cursor={last_timestamp}&size={size} interface. The backend service queries data where timestamp < last_timestamp each time, preventing duplicate data.

How to Handle Peak Traffic?

Handling peak traffic is a common challenge for any large-scale system. Here are some strategies:

  • Queueing: Place excess requests in a message queue so they can be processed when resources become available.
  • Rate Limiting & Throttling: Control how frequently a user, client, or IP address can send requests.
  • Auto-scaling: Automatically scale server and database capacity based on traffic load.
  • Autoscaling Load Balancing: Adjust load balancer capacity to distribute network traffic efficiently.
  • Caching: Reuse stored request results to reduce database pressure.

How to Handle Spam Comments?

We can use a spam detection system that uses machine learning algorithms to identify and block spam comments. The Spam Check can be integrated as an independent service, with the addition of a spam_comments collection to the Comment Database.

When new comments are identified as spam, they are not saved in the main comments collection. Instead, they are saved in the spam_comments collection so administrators can review, label them as non-spam, and move them back to the comments collection.
