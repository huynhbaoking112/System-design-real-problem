Part 1

Problem Statement

Every time a user plays a song, the service records a play event. The goal is to expose the top K most played songs. The lesson starts with all-time rankings, then increases complexity with high throughput and sliding time windows.

Global windowTop K across all historical play events.
Sliding windowTop K inside the last X minutes, hours, or days.
High QPSPartition stream events and merge local rankings.

Part 2

Requirements

Functional

  • Return the top K most played songs of all time on a single instance.
  • Return the all-time top K when play traffic is high and the system must scale out.
  • Return the top K songs in the last X time using a sliding window.

Scale and quality

  • 10 billion song plays per day.
  • 100 million songs.
  • Top K API receives about 1 million requests per day.
  • K is usually between 100 and 1000.
  • Low latency and high accuracy are preferred.

Part 4

Global Window, Low QPS

A single instance can maintain all-time top K with a hash map for counts and a min heap for ranking. The heap head is the least played song among the current top K.

How the single-instance version works

In the all-time version, a song's count only moves upward. That makes the data structure simple: the hash map is the source of truth for counts, and the min heap is just a compact index of the current top K songs.

  • song_counts stores total plays per song.
  • The min heap stores the current top K candidates.
  • The hash map can store heap index metadata to update heap entries in O(log K).

Example update

If songA receives a new play, the service first increments song_counts[songA]. If songA is already in the heap, we update its heap entry and re-heapify. If it is not in the heap, we compare its count with the heap root. Only when it beats the current smallest top-K item do we replace the root. 

song_counts[songA] += 1

if songA is in heap:
  update heap node and reheapify
else if song_counts[songA] > heap.min.count:
  replace heap.min with songA

Part 5

Global Window, High QPS

To scale throughput, partition events by song_id. This keeps all plays for the same song on the same worker, avoiding split counts and simplifying local aggregation.

This section is still solving the all-time Top K problem. The difference from the previous section is traffic: one machine cannot process all play events, so we divide the stream into partitions and let many workers maintain local rankings in parallel.

Play Events Stream hash(song_id) Workers Coordinator Cache

Worker responsibility

  • Consume assigned stream partitions.
  • Maintain local song counts.
  • Maintain local top K for its partition.

Coordinator responsibility

  • Collect local top K from workers.
  • Merge num_workers * K candidates.
  • Publish global top K to cache for fast API reads.

Why partition by song_id?

If events are distributed round-robin, the same song can be processed by multiple workers. Then no worker knows the full count for that song. By hashing song_id, every play for the same song always lands on the same worker, so each worker has exact local counts for its assigned songs. 

partition = hash(song_id) % num_partitions

songA -> worker 1
songB -> worker 3
songC -> worker 2

How global Top K is produced

Each worker exposes only its local top K. The coordinator does not need to scan every song in the system. It merges the candidates from all workers and keeps the highest K items globally. 

worker_1_top_k + worker_2_top_k + ... + worker_n_top_k
  -> coordinator merge
  -> global top K
  -> Redis cache
  -> API reads cached result

Snapshot Recovery

Since heaps and hash maps live in memory, each worker periodically snapshots its state and stream offset. After a crash, the worker restores the snapshot and replays events from the saved offset.

What is stored in a snapshot?

  • Heap data: the current local top K state for that worker.
  • Count map: current counts for songs owned by that worker.
  • Stream offset: the last event position already processed by the worker.

Recovery flow

When a worker restarts, it loads the newest snapshot, restores the in-memory structures, and resumes consuming from the stored offset. If some events arrived after the snapshot was written, replaying from that offset catches the worker up without losing counts. 

load latest snapshot
restore song_counts and heap
seek stream consumer to saved offset
replay missed events
resume normal processing

Part 6

Sliding Window Top K

Sliding windows are harder because counts both increase and decrease. New events enter the window, while old events leave the window and must be subtracted.

For example, with a 1-hour window, at 10:00 we count events in [09:00, 10:00]. At 10:01 the window becomes [09:01, 10:01]. Events from 10:00 to 10:01 are added, and events from 09:00 to 09:01 are removed.

End offset

for event in new_events:
  count[event.song_id] += 1

Beginning offset

for event in expired_events:
  count[event.song_id] -= 1

Why a plain min heap is not enough

In all-time Top K, counts only increase, so a size-K min heap works well. In a sliding window, counts can decrease when events expire. If a song currently in top K drops, a song outside the heap may need to enter, but a size-K heap does not know which outside song is next. For exact sliding-window rankings, the worker needs a ranking structure over all active songs in the window, not only the current K songs.

Event-level offset trade-off

The two-offset model is exact, but it may be expensive. If one minute contains millions of expired play events, the beginning offset must iterate through all of those events and decrement counts one by one. This is why the bucket-based approach below is usually easier to operate.

Part 7

Bucket-Based Exact Implementation

A more practical implementation aggregates events into fixed time buckets. For a 1-hour window with 1-minute precision, each worker keeps 60 circular buckets. Each bucket is a hash map of per-song counts for that minute.

The key idea is that the worker does not need to remember every raw event forever. It only needs enough information to subtract the bucket that just expired. Therefore each bucket stores song_id -> count_in_that_bucket, not just one total number. 

time_buckets = array[60] of HashMap<song_id, count_in_bucket>
song_counts = HashMap<song_id, current_window_count>
count_to_songs = TreeMap<count, Set<song_id>>

Time buckets

10:01songA: 3, songC: 4
10:02songB: 10

Current counts

songA8
songB12
songC4

TreeMap ranking

12{songB}
8{songA}
4{songC}

What each structure is responsible for

Time buckets

Tell us which song counts should be removed when an old minute leaves the sliding window.

Song counts

Tell us the current total count of each song inside the full active window.

TreeMap ranking

Lets us read songs from highest count to lowest count without sorting every song on each request.

When an event arrives

The worker updates three places. First it increments the current minute bucket. Then it increments the total window count. Finally it moves the song from its old count bucket to its new count bucket in the TreeMap. 

current_bucket[song_id] += 1

old_count = song_counts[song_id]
new_count = old_count + 1

remove song_id from count_to_songs[old_count]
add song_id to count_to_songs[new_count]
song_counts[song_id] = new_count

When a bucket expires

When the clock moves forward, the circular array slot for the oldest minute is reused. Before reusing it, the worker reads every song in that bucket and subtracts those aggregate counts from the active window. 

for song_id, expired_count in expired_bucket:
  old_count = song_counts[song_id]
  new_count = old_count - expired_count

  remove song_id from count_to_songs[old_count]

  if new_count > 0:
    add song_id to count_to_songs[new_count]
    song_counts[song_id] = new_count
  else:
    remove song_id from song_counts

Concrete 3-minute example

Suppose the active window has three buckets: 10:00, 10:01, and 10:02. The current totals are songA = 8, songB = 12, and songC = 4. When time moves to 10:03, bucket 10:00 expires. If bucket 10:00 contains songA: 5 and songB: 2, the worker updates totals to songA = 3 and songB = 10, then updates the TreeMap ranking accordingly.

To read local top K, the worker scans the TreeMap from highest count to lowest count and stops after K songs. This avoids sorting every active song on every query. 

for count in count_to_songs.descending_keys():
  for song_id in count_to_songs[count]:
    result.append({ song_id, count })
    if len(result) == K:
      return result

Important trade-off

This design is exact, but it keeps state for active songs in each worker. If the number of active songs is huge, memory usage can be high. If we can tolerate small errors, approximate algorithms such as Count-Min Sketch or Space-Saving can reduce memory pressure.

Part 8

Approximate Results

If exact rankings are not required, approximate structures such as Count-Min Sketch can reduce memory and update cost. This is useful when near-real-time trends matter more than exact counts.

  • Updates and queries are close to constant time.
  • Memory usage is lower than exact per-song state.
  • Counts are upper-bound estimates, so rankings can contain small errors.
